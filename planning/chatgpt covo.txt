Positioning Doc
1. Positioning statement

ExecutionKit is a minimal Python library for composable LLM reasoning patterns.
It gives developers a small set of reusable execution primitives — consensus, refine loops, ReAct tool loops, and pattern composition with shared budget tracking — that work across any OpenAI-compatible endpoint using one simple provider class and no SDK dependencies beyond stdlib HTTP/JSON plus the package’s tiny type/runtime footprint.

2. Market position
What ExecutionKit is
A pattern library, not a full agent platform.
An execution layer, not a gateway or proxy.
A small orchestration helper, not a stateful graph runtime.
A budget-aware composition toolkit, not an observability suite.
A portable OpenAI-compatible client abstraction, not a native adapter matrix for every provider.
What ExecutionKit is not
Not a LangGraph-style framework for durable, long-running, stateful workflows. LangGraph explicitly targets those use cases.
Not a PydanticAI-style full agent framework with broad provider-native support and tool ecosystems. PydanticAI explicitly positions itself as model-agnostic with built-in support for many providers and tool modes.
Not a LiteLLM-style gateway or unified LLM control plane with spend dashboards, budgets, routing, and logging. LiteLLM explicitly positions around unified access, proxying, and spend tracking.
Not an OpenAI Agents SDK-style multi-agent orchestration package. The Agents SDK explicitly centers agents, tools, and orchestration flows.
3. Differentiators
Primary differentiators

Pattern-first API
The user-facing abstraction is a reasoning pattern, not an agent, graph, router, or workflow engine. That is the cleanest conceptual separation from adjacent tools.

Shared budget tracking across composition
Budget tracking is part of execution semantics, not just reporting. In your plan, pipe() shares remaining budget across steps and costs merge through PatternResult, which is more specific than generic usage logging.

OpenAI-compatible by design
The product boundary is deliberate: support any endpoint that speaks OpenAI-style chat completions, rather than taking on every native provider SDK. That makes the abstraction easier to explain and lighter to maintain.

Zero-SDK philosophy
This is emotionally and practically valuable. In a market full of frameworks, wrappers, SDK layers, gateways, and adapters, “URL + API key + JSON” is a real positioning asset.

Secondary differentiators

Portable across cloud and local serving stacks
Because the boundary is OpenAI-compatible HTTP, the same library can target OpenAI, Azure-style endpoints, Ollama, LM Studio, vLLM, and similar servers without changing the core pattern API. That is narrower than “all providers,” but sharper and more honest. LM Studio, Ollama, and vLLM all document OpenAI-compatible serving paths.

ExecutionKit is for developers who:

already know how to make a basic LLM call,
do not want a full agent framework,
want a few reusable higher-level reasoning patterns,
care about portability across endpoints,
want to keep cost/budget visible during chained execution,
prefer small abstractions over large ecosystems.

Patterns
consensus()

Use when a single generation is not enough and agreement matters.

result = await consensus(
    provider,
    "Summarize the issue and assign a severity.",
    num_samples=5,
)

Returns a PatternResult[str] with metadata such as agreement ratio, unique responses, and tie count.

refine_loop()

Use when quality improves through repeated critique and revision.

result = await refine_loop(
    provider,
    "Draft a short customer-facing explanation of this outage.",
    target_score=0.9,
    max_iterations=5,
)

Uses convergence detection, evaluator scoring, and stopping rules to decide when to stop refining.

react_loop()

Use when the task needs tools and bounded step-by-step reasoning.

result = await react_loop(
    provider,
    "Find the latest failed deployment and summarize root cause.",
    tools=my_tools,
    max_rounds=8,
)

Supports tool schemas, bounded rounds, tool timeouts, and usage tracking throughout the loop.

pipe()

Use when you want to chain reasoning patterns into one execution flow.

result = await pipe(
    provider,
    "Analyze this incident and prepare a final response.",
    consensus,
    refine_loop,
    max_budget=TokenUsage(input_tokens=20000, output_tokens=8000, llm_calls=20),
)

pipe() merges usage and passes the remaining budget forward through each step.

ExecutionKit v0.1 – Initial Documentation and Updated Specification Plan

ExecutionKit is a minimal Python library designed for pattern‑first execution over any endpoint that implements the OpenAI‑compatible /chat/completions API. It provides a small set of reusable reasoning patterns with built‑in budget awareness and a lightweight provider abstraction. This document summarises the core concepts of the library, outlines the features that ship in v0.1, notes what is deferred to v0.2, and lays out an updated specification plan to keep the scope tightly focused.

Key Concepts
Pattern‑First Execution: ExecutionKit packages three high‑level reasoning workflows—consensus, refine_loop, and react_loop—as first‑class functions. Instead of building agents or workflows from scratch, developers can call these patterns to sample from multiple models, iteratively refine answers, or drive a bounded ReAct‑style tool loop.
Budget‑Aware Composition: The library tracks token usage and call counts for each pattern. Using pipe() you can chain patterns together while sharing a TokenUsage budget, so you always know how much input, output, and call quota remains.
OpenAI‑Compatible Transport: A single Provider class wraps HTTP interactions with /chat/completions endpoints. The provider takes a base URL, model name, and optional API key, then sends JSON payloads over the standard interface. This design works seamlessly with OpenAI, Azure, Groq, Together, Ollama, LM Studio and any local or cloud server that advertises OpenAI compatibility.
Minimal Dependencies: ExecutionKit uses only the Python standard library for HTTP and JSON plus Pydantic for type validation. There are no heavy provider SDK dependencies.
What Ships in v0.1
Patterns: consensus, refine_loop, react_loop
Composition: pipe() helper to chain patterns with a shared budget
Session: Kit for managing pattern execution
Provider: a single universal Provider class (URL + model + API key)
Engine: retry logic, parallel execution helpers, convergence detection, and a JSON extraction fallback for models that return ill‑formed JSON
Types: PatternResult[T], TokenUsage, Tool, ToolCall, VotingStrategy (majority and unanimous), Evaluator, and LLMResponse
Errors: ExecutionKitError hierarchy covering rate limits, provider failures, budget exhaustion, consensus failures and max‑iteration conditions
Features: CostTracker, checked_complete, validate_score, agreement_ratio, was_truncated, and a ConvergenceDetector
Infrastructure: CLI, pyproject.toml, examples, a README and CONTRIBUTING guide
Deferred to v0.2
Tree‑of‑thought / beam search
Native Anthropic provider and message conversion
TraceEntry and ProgressCallback (observability hooks)
StreamingProvider for real‑time output
OpenTelemetry integration

Updated Specification Plan

The crowded LLM tooling space already includes full agent frameworks, gateways, and orchestration runtimes. ExecutionKit differentiates itself by staying small and opinionated. To preserve that identity:

v0.1 remains tightly scoped. Only the core patterns (consensus, refine_loop, react_loop), the pipe() composition helper, the Provider abstraction and essential engine utilities ship. There is no graph runtime, multi‑agent handoff, or provider‑specific adapter matrix. Observability and native Anthropic support are explicitly deferred.
v0.2 enhancements focus on composition, not scope creep. The next release should add richer composition ergonomics (policy objects for retries and budgets), optional progress callbacks, lightweight trace objects, streaming support, and optionally separate provider implementations via the LLMProvider protocol. Graph runtimes, dashboards, governance features, and broad provider sprawl remain out of scope.

By following this plan, ExecutionKit stays a focused execution library rather than an orchestration framework. Its pattern‑first design and budget‑aware composition complement rather than compete with more complex agent platforms.

Hero Example

Here is a concise example using consensus with a local model served by Ollama:

from executionkit import Provider, consensus

# Connect to a local Ollama server that exposes an OpenAI‑compatible API
provider = Provider("http://localhost:11434/v1", model="llama3.2")

result = await consensus(
    provider,
    "Classify this support ticket into billing, bug, or access.",
    num_samples=5,
)

print(result.value)  # The chosen classification
print(result.cost)   # TokenUsage(input_tokens=250, output_tokens=45, llm_calls=5)
print(result.metadata["agreement_ratio"])  # e.g., 0.8 (4 out of 5 agreed)

The same code can run against any OpenAI‑compatible endpoint—cloud or local—by changing the provider’s base URL and model name.

Alternatives to consider

ExecutionKit focuses on pattern‑first execution with minimal dependencies. If you later decide you need more orchestration features or different abstractions, there are several open‑source frameworks worth evaluating:

LangGraph extends LangChain into a directed acyclic graph (DAG) architecture. Each node is a step and edges control data flow, making it suitable for stateful workflows where you need precise branching and error handling.
OpenAI Agents SDK packages OpenAI’s capabilities into a structured runtime for multi‑step and multi‑agent tasks. It provides a familiar prompt interface and is deeply integrated with OpenAI models.
Google Agent Development Kit (ADK) offers built‑in multi‑agent orchestration and session management; it’s a good fit if you’re invested in Gemini models or Google’s AI ecosystem.
Smolagents takes a radically simple, code‑centric approach: agents generate and execute Python code in a minimal loop. This is ideal when you want fast setup and lightweight ReAct‑style prompting without complex graphs.
CrewAI focuses on role‑based collaboration. It organizes agents into a “crew,” each with a specialized role, and coordinates their workflow for multi‑agent tasks.
AutoGen (Microsoft Research) frames workflows as asynchronous conversations among agents, which is useful for long-running tasks and dynamic dialogues.
Semantic Kernel is a .NET‑first library designed to orchestrate AI “skills” and traditional code, with an emphasis on enterprise integration and security.
How to prepare
Install ExecutionKit and dependencies: Ensure you have Python 3.10+, install the library via pip install executionkit, and verify Pydantic is installed.
Set up local or cloud models: For local models, run an OpenAI‑compatible server (e.g. Ollama, LM Studio, or vLLM). For cloud, have API keys ready for OpenAI, Groq, Together, or Azure endpoints. The Provider class in ExecutionKit only needs a base URL, model name, and API key.
Define your budget policies: Decide on token budgets and cost limits for your patterns. ExecutionKit lets you enforce budgets via TokenUsage and pipe() so costs don’t exceed your constraints.
Organize documentation: Import the provided Markdown file into Notion or another documentation system. Use it as your “single source of truth,” updating it as the spec evolves.
Evaluate the alternatives: If your needs go beyond pattern‑first execution—such as full graph-based orchestration, multi-agent collaboration, or enterprise integration—compare the frameworks above. LangGraph suits DAG workflows; the OpenAI Agents SDK and Google ADK provide structured runtimes; smolagents and CrewAI offer lightweight approaches.

ExecutionKit v0.1 – Initial Documentation and Updated Specification Plan

ExecutionKit is a minimal Python library designed for pattern‑first execution over any endpoint that implements the OpenAI‑compatible /chat/completions API. It provides a small set of reusable reasoning patterns with built‑in budget awareness and a lightweight provider abstraction. This document summarises the core concepts of the library, outlines the features that ship in v0.1, notes what is deferred to v0.2, and lays out an updated specification plan to keep the scope tightly focused.

Key Concepts
Pattern‑First Execution: ExecutionKit packages three high‑level reasoning workflows—consensus, refine_loop, and react_loop—as first‑class functions. Instead of building agents or workflows from scratch, developers can call these patterns to sample from multiple models, iteratively refine answers, or drive a bounded ReAct‑style tool loop.
Budget‑Aware Composition: The library tracks token usage and call counts for each pattern. Using pipe() you can chain patterns together while sharing a TokenUsage budget, so you always know how much input, output, and call quota remains.
OpenAI‑Compatible Transport: A single Provider class wraps HTTP interactions with /chat/completions endpoints. The provider takes a base URL, model name, and optional API key, then sends JSON payloads over the standard interface. This design works seamlessly with OpenAI, Azure, Groq, Together, Ollama, LM Studio and any local or cloud server that advertises OpenAI compatibility.
Minimal Dependencies: ExecutionKit uses only the Python standard library for HTTP and JSON plus Pydantic for type validation. There are no heavy provider SDK dependencies.
What Ships in v0.1
Patterns: consensus, refine_loop, react_loop
Composition: pipe() helper to chain patterns with a shared budget
Session: Kit for managing pattern execution
Provider: a single universal Provider class (URL + model + API key)
Engine: retry logic, parallel execution helpers, convergence detection, and a JSON extraction fallback for models that return ill‑formed JSON
Types: PatternResult[T], TokenUsage, Tool, ToolCall, VotingStrategy (majority and unanimous), Evaluator, and LLMResponse
Errors: ExecutionKitError hierarchy covering rate limits, provider failures, budget exhaustion, consensus failures and max‑iteration conditions
Features: CostTracker, checked_complete, validate_score, agreement_ratio, was_truncated, and a ConvergenceDetector
Infrastructure: CLI, pyproject.toml, examples, a README and CONTRIBUTING guide
Deferred to v0.2
Tree‑of‑thought / beam search
Native Anthropic provider and message conversion
TraceEntry and ProgressCallback (observability hooks)
StreamingProvider for real‑time output
OpenTelemetry integration
