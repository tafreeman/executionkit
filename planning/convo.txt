● Great question. Let me explain how ExecutionKit plugs into LLM calls.

  How the Provider Protocol Works

  ExecutionKit does NOT ship with any LLM provider. Instead, the user wraps their existing client in a 10-line adapter:

  Example: Wrapping OpenAI

  from openai import AsyncOpenAI
  from executionkit import LLMProvider, LLMResponse

  class OpenAIProvider:
      def __init__(self, model: str = "gpt-4o"):
          self.client = AsyncOpenAI()
          self.model = model

      async def complete(self, messages, *, temperature=0.7, max_tokens=4096,
                         tools=None, response_format=None, **kwargs) -> LLMResponse:
          params = {"model": self.model, "messages": messages,
                    "temperature": temperature, "max_tokens": max_tokens}
          if tools:
              params["tools"] = tools
          if response_format:
              params["response_format"] = {"type": "json_schema",
                                            "json_schema": response_format.model_json_schema()}

          resp = await self.client.chat.completions.create(**params)
          choice = resp.choices[0]
          return LLMResponse(
              content=choice.message.content or "",
              tool_calls=[tc.model_dump() for tc in (choice.message.tool_calls or [])],
              input_tokens=resp.usage.prompt_tokens,
              output_tokens=resp.usage.completion_tokens,
              model=self.model,
          )

  Then use any pattern:

  provider = OpenAIProvider("gpt-4o-mini")

  # Tree of Thought
  result = await tree_of_thought(provider, "Solve: make 24 from 1,3,4,6")

  # ReAct with tools
  result = await react_loop(provider, "Research topic X", tools=[web_search])

  # Consensus voting
  result = await consensus(provider, "Classify: ...", num_samples=5)

  How it flows inside a pattern:

  tree_of_thought(provider, prompt)
    │
    ├── Branch generation:
    │     provider.complete([{"role": "user", "content": "Generate 3 approaches to: {prompt}"}])
    │     → LLMResponse(content="Approach 1: ...\nApproach 2: ...\nApproach 3: ...")
    │     → Parse into 3 Branch objects
    │
    ├── Branch evaluation (parallel via gather_bounded):
    │     provider.complete([{"role": "user", "content": "Score this approach 0-1: {branch.content}"}])
    │     → LLMResponse(content="0.85")  → branch.score = 0.85
    │     (repeated for each branch, concurrently)
    │
    ├── Prune: keep top beam_width branches (e.g., 2 of 3)
    │
    ├── Recurse: for each surviving branch, generate sub-branches at next depth
    │
    └── Return: PatternResult(value=best_leaf.content, score=0.92, cost=CostMetrics(...))

  Key design: Provider is a Protocol, not a base class

  @runtime_checkable
  class LLMProvider(Protocol):
      async def complete(self, messages, **kwargs) -> LLMResponse: ...

  This means any object with a complete() method works — no inheritance required. Users can wrap:
  - openai.AsyncOpenAI
  - anthropic.AsyncAnthropic
  - litellm.acompletion
  - ollama REST calls
  - A mock for testing

  The pattern functions call provider.complete() with standard chat messages. They never touch HTTP, API keys, or provider-specific
  logic. That's entirely the user's responsibility.

  What about structured output?

  When a pattern needs structured data (e.g., ToT branch scores, ReAct tool calls), it passes response_format=SomeModel to
  provider.complete(). If the user's provider supports native structured output (OpenAI, Anthropic), their adapter uses it. If not,
  ExecutionKit falls back to the existing extract_json() balanced-brace parser from the codebase.
